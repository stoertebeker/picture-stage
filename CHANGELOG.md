# Changelog

All notable changes to Picture-Stage are documented in this file.

## [Unreleased]

### Security

- **A disabled photographer's share links stop working for guests** (`picture-stage-cxs`). Previously the guest token resolver checked only the share-token hash and the gallery status, never the owner's account status — so locking a photographer (`status=disabled`) left every share link of their galleries fully usable (guests could still view, select and comment). Both token resolvers (the HTML viewer and the JSON API) now join the owner and require a login-allowed status, returning the same neutral `404 Gallery not found` as an unknown token. The gate lives in the resolver, so no share sessions are mutated and re-enabling the account restores access automatically.
- **Gallery password verification via the web form is now rate-limited** (`picture-stage-a2d`). The HTML endpoint `POST /g/{token}/verify-password` had no rate limit (its JSON-API counterpart always had 5/minute), allowing unthrottled brute-force attempts against the bcrypt gallery password through the guest form. Both paths now enforce 5/minute.
- **CSP hardened: `'unsafe-eval'` removed from `script-src`** (`picture-stage-u3s`). The standard Alpine.js build compiles inline expressions with the `Function` constructor, which forced `'unsafe-eval'` in the Content-Security-Policy (a broad XSS-amplification surface). All inline Alpine expressions were migrated to `Alpine.data()` component registrations plus a delegated dialog/listener pattern in `app.js` (data-attributes instead of inline `$refs.x.showModal()` / `$el.closest('dialog').close()` / `hx-on`), and the build was switched to `@alpinejs/csp`, which evaluates expressions without `eval`. `script-src` is now `'self'` only — defense-in-depth against XSS.
- **Signup no longer reveals whether an account exists** (`picture-stage-42q`). Previously a repeated registration with an already-known email returned HTTP 409 with an explicit hint (account enumeration / information disclosure), on both the web form and the JSON API. A signup with an existing email (as user *or* pending) now returns the same neutral success response as a fresh one, creates no new pending signup, and never overwrites an existing account/password (which would otherwise be an account-takeover vector). Best-effort timing equalization (matching bcrypt cost) reduces the existing-vs-new timing signal.
  - **BREAKING (API):** `POST /api/v1/auth/signup` no longer returns `verification_token` in the response body — emitting it for fresh signups but `null` for existing emails would itself leak existence. The token is persisted server-side and is delivered by email (`picture-stage-x8t`). Clients that read the token from the response must change.
- **Stateless JWTs are invalidated on password reset and account lock** (`picture-stage-7kr`). Previously an already-issued access token stayed valid for up to 24 h after an admin reset a user's password or disabled the account. Tokens now carry an `iat` claim and are rejected once the per-user `tokens_valid_after` cut-off (set on reset/lock) is passed.
  - No mass-logout on upgrade: the cut-off defaults to NULL, so existing tokens keep working until the next reset/lock of that user.
  - The check compares server-side timestamps only, so client clock skew is irrelevant. Multi-instance deployments should keep server clocks in sync (NTP).
- **Share links are now always HTTPS in production** (`picture-stage-0hp`). Behind a TLS-terminating reverse proxy the container only sees plain HTTP, which caused share links to leak the replayable share token over `http://`. Links are now built from the configured `APP_URL` and the scheme is forced to `https://` in production.
  - **Action required:** set `APP_URL` to your public HTTPS domain (e.g. `https://photos.example.com`). A missing/default `APP_URL` falls back to the request host with the scheme corrected to https.

### Changed

- **Guest expired page redesigned to Editorial Dark** (`picture-stage-9ql`, v0.6). The "gallery expired" dead-end (HTTP 410) still used the legacy gray/white styling with obsolete `dark:` utilities and inherited the legacy `guest_base` shell. It now extends `auth_base.html` — the same Editorial-Dark chrome as the auth pages (`gmo`) — so it carries the brand wordmark, theme toggle and legal footer and is fully token-styled (theme-safe in dark and light). The 410 status and the i18n copy are unchanged.
- **Guest gallery grid loads progressively** (`picture-stage-am9`). The guest selection view rendered every image as a grid DOM item up front, so a large gallery shipped a huge initial HTML document (~630 KB at 122 images, growing linearly) and mounted all items at once — leaving the page slow to load and unresponsive while it did (`loading="lazy"` only deferred the image *downloads*, not the HTML/DOM cost). The grid now renders the first 30 items and loads the rest on scroll via an HTMX `revealed` sentinel, roughly halving the initial payload and keeping the mounted-item count constant regardless of gallery size. The full image list still ships in `data-images` for the lightbox/selection state. Grid bindings were moved from array indices to image ids (with an `id→image` map for O(1) lookups), which also fixes a latent bug where index-based bindings pointed at the wrong images under an active filter/sort. `decoding="async"` added to grid images.
- **Re-opening a finished gallery is no longer mislabelled "Share"** (`picture-stage-8wv`). On the gallery detail page the status transition `completed → shared` (and `archived → shared`) re-opens the gallery for the model, but the button reused the generic `transition_shared` label ("Teilen"/"Share"), which wrongly suggested creating a share link. These re-activating transitions now use a dedicated `transition_reopen` label ("Erneut öffnen"/"Reopen"). In addition the status badge moved from above the title into the action-button group, so the current state and the buttons that change it read as one unit. Label/layout only — no change to the allowed transitions or backend behaviour.
- **Auth pages redesigned to Editorial Dark** (`picture-stage-gmo`, v0.6). Login, signup and verify moved off their legacy gray/white styling onto the Editorial-Dark token system. A new shared `auth_base.html` layout centralises the page chrome (head, fonts, skip link, theme toggle, brand wordmark, legal footer, scripts) that the standalone pages previously duplicated, so each page now only carries its card. The legal footer (`_legal_footer.html`) moved from a fixed overlay into the normal flow with a sticky-footer layout, so a taller card can no longer overlap it (setup page adjusted to the same layout). No behavioural change — forms, CSRF and routes are untouched.
- **Cookie consent banner removed** (`picture-stage-a15`). The site sets only strictly necessary cookies — `session` (login JWT) and `csrf_token` — plus a language-preference cookie (`lang`) and a `theme` entry in `localStorage`; none of these require consent under GDPR/TTDSG §25(2). The previous banner was an accept-only, no-op element (it gated no cookie, offered no rejection) — more of a liability than a safeguard. It was removed from both layouts along with its Alpine component and i18n strings. The necessary cookies are now documented in a privacy-policy template (`legal/datenschutz.example.md`) for operators to fold into their own `datenschutz.md`.
- **BREAKING:** `WATERMARK_OPACITY` environment variable now expects **float (0.0-1.0)** instead of **int (0-255)**
  - Old format (v0.0): `WATERMARK_OPACITY=255` (opaque) → `WATERMARK_OPACITY=1.0`
  - Old format (v0.0): `WATERMARK_OPACITY=0` (transparent) → `WATERMARK_OPACITY=0.0`
  - Old format (v0.0): `WATERMARK_OPACITY=128` (50% opaque) → `WATERMARK_OPACITY=0.5`
  - Aligns with CSS/HTML5 opacity conventions (0.0-1.0)
  - Gallery-level `watermark_config.opacity` already uses float (0.0-1.0)
  - Conversion to PIL alpha (0-255) happens internally in `app/images/processing.py`

### Fixed

- **Mobile lightbox swipe hint no longer lingers** (`picture-stage-jwc`). In the guest selection view the "swipe" hint pill at the bottom of the mobile lightbox was bound only to `images.length > 1`, so it stayed on screen the whole time. It now appears when a photo is opened and fades out after ~3.5 s (Alpine state `showSwipeHint` set in `openLightbox()` with a self-clearing timer), reappearing each time a photo is opened. Mobile-only (`sm:hidden`); no change to swipe behaviour itself.
- **Lightbox toolbar no longer covers the image or its watermark** (`picture-stage-0d1`). In the guest selection view, the bottom control bar (select/favorite/comment) was an `absolute bottom-0` overlay on top of the vertically-centered image, so on taller images it hid the lower edge — including the bottom-right watermark. The lightbox is now a flex column: the image sits in a `flex-1` stage and the toolbar in normal flow beneath it, so they never overlap. The image is fitted via `object-contain` inside the stage (an absolute fill, because a percentage `max-height` on a flex child is not honoured in nested flex columns), keeping tall and wide images fully visible at any viewport height. No JS changes.
- **The "complete review" button works again from the browser** (`picture-stage-4gr`). `POST /g/{token}/complete` declared `session_id` as a bare `uuid.UUID`, which FastAPI binds as a *query* parameter — but the frontend JS (`completeReview()`) sends it in the JSON body. The mismatch returned `422`, and because the JS never checked the response status (it just reloads), the failure was silent: galleries were never transitioned to `completed` via the web UI, the gallery-wide read-only gate never engaged, and the new owner completion email (`picture-stage-16l`) never fired. The endpoint now reads `session_id` from a `CompleteReviewRequest` body model, matching the client. Regression tests pin the body contract.
- **Legal notice and privacy policy now reachable from the public auth pages** (`picture-stage-aqn`). The Impressum/Datenschutz footer links lived only in the logged-in (`base.html`) and guest (`guest_base.html`) layouts. The standalone login, signup, verify and setup pages are separate documents and carried no link, so a visitor without an account had no way to reach them (the pages themselves were always public) — falling short of the "directly reachable" requirement (DDG §5 / GDPR Art. 13). A shared `_legal_footer.html` partial with both links is now included on all four pages.
- **Guest grid selection/favorite UI restored under the CSP build** (`picture-stage-2gb`). The `@alpinejs/csp` build cannot parse optional chaining (`images[N]?.selected`), so the seven affected grid bindings threw at runtime (112 console errors) and selection rings, check badges, hover toggles and favorite hearts never rendered. The grid now calls null-safe `isSelected(idx)`/`isFavorited(idx)` helpers on the `guestViewer` component instead. Live prod-verified.
- **Status-color opacity modifiers now render** (`picture-stage-toj`). Tailwind 3.4 cannot compose opacity modifiers (`bg-status-danger/10`, `border-status-danger/40`) from hex-based `var()` color tokens — the classes were never generated, so the guest password-gate error alert and `form_error` had no red tint. The four `status-*` tokens are now defined as space-separated RGB components with `rgb(var(--color-status-X) / <alpha-value>)`. Also fixes the `form_error` `text-red-300` legacy (unreadable in light mode). Live prod-verified in both themes.
- **Upload progress no longer breaks the CSP parser** (`picture-stage-3uh`). The inline handler `@htmx:after-request.window="uploadProgress = 0; uploading = false"` is a multi-statement expression the `@alpinejs/csp` build cannot parse (`Unexpected token: uploading`), so the upload-progress state never reset. The reset is now wrapped in `uploadZone.onUploadComplete()`. (Pending live prod-verification.)

### Added

- **Selection result view with a Lightroom-ready filename export** (`picture-stage-r84`). A new owner-only page `/galleries/{id}/selection` lists every image the model marked (selected *or* favorited) with a thumbnail, the filename and per-image badges, linked from the gallery detail page once the gallery is `shared`/`completed`. It offers a "copy for Lightroom" button plus `.txt`/CSV downloads, all scoped to the new `filter=marked` export filter. The download adds a `format=txt` export: comma-separated filenames *with extension* on a single line — the format Lightroom Classic's filename filter and Capture One's "Select by filename" expect for a paste-in selection (research showed one-name-per-line does **not** work directly in Lightroom). The copy payload is built server-side so clipboard and `.txt` match exactly; filenames ride a `data-*` attribute (Jinja-autoescaped) and render via Alpine `x-text`, never evaluated as code.
- **Full-screen image lightbox in the photographer's gallery** (`picture-stage-x4o`). Clicking an image in the gallery detail grid previously opened a single-image modal with no navigation. It now opens a full lightbox mirroring the guest viewer — arrow buttons, keyboard ←/→/Esc, mobile swipe and adjacent-image preload — but read-only (no select/favorite/comment controls; the photographer only views). Image data rides a `data-images` attribute (ready images only) and the grid opens by image id; the user-controlled filename renders via Alpine `x-text`, keeping it XSS-safe under the `'self'`-only CSP.
- **Email alert to the photographer when a model completes a review** (`picture-stage-16l`). When a model finishes reviewing a gallery (`POST /g/{token}/complete`), the gallery owner now receives an email summarising the result (image count, selected, favorited, dashboard link). The `gallery_completed` mail template and the completion trigger already existed but were gated behind the per-user `NotificationConfig` opt-in, for which there is no UI — so nothing was ever delivered. A new config-free `notify_owner_gallery_completed()` (mirroring `notify_admins_signup`) guarantees the alert, gated only by the new `NOTIFY_OWNER_ON_COMPLETION` setting (default true) plus a configured `SMTP_HOST`. The recipient is always the owner's own address (loaded from the DB), and delivery failures are logged without ever aborting the completion. The pre-existing config-gated `send_notification` path is kept for webhooks / a future notification UI.
- **Optional gallery message shown to the model** (`picture-stage-dii`). The gallery detail page has a new "message to the model" panel: a free-text note (max 1000 chars) the photographer can attach to a gallery. When set, it renders above the image grid in the guest viewer; when empty, no block appears. The note is added to a new nullable `galleries.guest_message` column (migration `0005`) and is exposed on the JSON API (`GalleryCreate`/`Update`/`Response`) and carried over on duplicate. The free text is autoescaped and rendered with `white-space: pre-line` (newlines preserved, no HTML passthrough) so a malicious note cannot inject markup into a guest's page.
- **Per-gallery watermark text + on/off toggle** (`picture-stage-bsr`). The gallery detail page now has a free-text watermark field and a "show watermark" checkbox in the watermark panel. Previously the preview watermark was hardcoded to `PREVIEW · {id}`; the upload pipeline (both the synchronous API path and the async preview worker) now resolves text/position/opacity/font-size from the gallery's own `watermark_config` (empty → global default, `{gallery_id}` placeholder supported) and skips the overlay entirely when disabled. The free text is control-char-stripped and capped at 200 chars. Applies to new uploads only — existing previews are not regenerated.
- **Configurable per-user gallery limit** (`picture-stage-5gi`). A new `MAX_GALLERIES_PER_USER` setting (default 5, `0` = unlimited) caps how many galleries a single user may own. Enforced via a shared helper (`app/galleries/quota.py`) on both creation paths — the JSON API returns `409 Conflict`, the HTMX frontend shows an error toast (i18n key `gallery.limit_reached`, DE/EN). The check runs only at creation time and never deletes or locks existing galleries, so adjusting the limit cannot strand a user already above it.
- **Gallery password can now be set, changed or removed after sharing** (`picture-stage-1y5`). Previously the password could only be chosen while creating the share link; changing it required revoking and re-creating the link, which rotated the token and invalidated the magic link already sent to the model. The share modal now offers a password section for active links, backed by `POST /galleries/{id}/password` (HTML frontend) — the share token stays untouched. Note: guests who already unlocked the gallery in an open tab keep their session; the gate applies from the next page load.
- **Guest password gate redesigned (Editorial Dark, `qdz.16`)**. The gate now shows the gallery name above the card, a friendlier error message and a privacy note, in dark and light theme. Verifying the password renders the full viewer page (the previous partial swap left the unlocked gallery without header, lightbox and selection state).
- **Verification email on signup** (`picture-stage-x8t`). A new registrant now receives an email with a confirmation link (built from `APP_URL`, HTTPS in production; the plaintext token travels in the link while the DB keeps only its SHA-256+salt hash). The mail is only sent on the genuine new-signup path, so it never fires for an already-registered email — keeping the account-enumeration guard (`picture-stage-42q`) intact. Config-free and gated by `SEND_VERIFICATION_EMAIL_ENABLED` (default true) plus a configured `SMTP_HOST`; delivery failures are logged and never abort a signup. Verification stays orthogonal to admin approval.
- v0.4 Frontend: Complete Jinja2 templates, HTMX router, Alpine.js interactivity, CSRF protection, Cookie-based auth, i18n (DE/EN)
- v0.3 Compliance: DSGVO pages + cookie banner, audit logging with CSV export, backup/restore CLI, encryption for sensitive fields
- v0.2 Features: Gallery lifecycle (draft → shared → completed → archived), expiry dates with guest enforcement, watermark customization
- v0.1 API: 27 REST endpoints, image upload/processing, selection event sourcing, share tokens, guest viewer

## [0.1.0] - Initial Release

- Minimal viable Picdrop alternative: photographers share galleries, models select/favorite images
- Self-hosted, zero-trust design: end-to-end encryption option, no external dependencies
- FastAPI backend + PostgreSQL + HTMX+Alpine frontend
- 181 tests (165 unit/security + 16 integration), CI/CD ready
